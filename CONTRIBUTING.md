# Contributing to VortexChat

## Prerequisites

- Node.js 22+, npm 10+ (this monorepo uses **npm** workspaces — do not use pnpm or yarn)
- The database is **SQLite** (via Drizzle ORM in `@vortex/db`) — no external
  database service needed for development
- Docker + Docker Compose (only for running the full self-hosted stack; see
  [`deploy/SELF-HOSTING.md`](./deploy/SELF-HOSTING.md))

---

## Branch naming

CI runs on `main`, `master`, and any branch matching `claude/**`.

```
claude/<short-description>-<id>
```

Examples:

```
claude/fix-rls-timeout-policy-9x3kA
claude/add-forum-channel-type-4z7mB
```

- Use kebab-case for the description
- The trailing ID is appended automatically by the Claude Code agent; human-authored
  branches may omit it
- Feature branches that do **not** match `claude/**` will skip CI — open a PR to
  `main` to trigger the full pipeline

## Commit message format

Follow [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>(<scope>): <short summary>
```

| Type | When to use |
|------|-------------|
| `feat` | New user-facing feature |
| `fix` | Bug fix |
| `perf` | Performance improvement |
| `refactor` | Code change with no functional effect |
| `test` | Adding or updating tests |
| `ci` | CI/CD changes |
| `docs` | Documentation only |
| `chore` | Dependency bumps, build tooling |

Scope is optional but encouraged — use the workspace or domain name:
`feat(dm)`, `fix(auth)`, `ci(migrations)`.

Keep the summary under 72 characters and written in the imperative mood
("add timeout column", not "added" or "adds").

## Database & migrations

The schema is defined with Drizzle ORM in `packages/db/src/schema/` and lives
in a single SQLite file. Migrations are generated by `drizzle-kit` and tracked
in `packages/db/migrations/` (SQL files + a `meta/_journal.json` journal).

### 1. Create / update your local database

```bash
# Applies all generated migrations + the FTS5 triggers to the SQLite file.
# Defaults to packages/db/data/vortex.db; override with DATABASE_URL=file:...
npm run db:migrate --workspace=packages/db
```

### 2. Writing a schema change

1. Edit the relevant table in `packages/db/src/schema/` (e.g. add a column).
2. Generate a migration from the schema diff:
   ```bash
   npm run db:generate --workspace=packages/db
   ```
   This writes a new `NNNN_<slug>.sql` under `packages/db/migrations/` and
   updates the journal + snapshot. **Never hand-edit the numeric prefix or
   reuse one** — `drizzle-kit` assigns the next index automatically.
3. Review the generated SQL, then apply it:
   ```bash
   npm run db:migrate --workspace=packages/db
   ```
4. If the change affects full-text search, update
   `packages/db/src/sql/fts5-and-triggers.sql` (applied after the generated
   DDL by `migrate.ts`; every statement must stay idempotent —
   `CREATE … IF NOT EXISTS`).

### 3. Verify migrations

```bash
npm run test:migration-smoke
```

This static-analysis script (`scripts/migration-smoke-test.mjs`) checks that
the Drizzle migration tree is internally consistent: filename format, no
duplicate version prefixes, and the journal matches the `.sql` files and
snapshots on disk. CI additionally applies the migrations to a throwaway
database to confirm they run cleanly.

## Running the full test suite locally

```bash
# Unit + integration tests (Vitest)
cd apps/web && npm test

# Type check all workspaces
npm run type-check

# Lint all workspaces
npm run lint

# Migration smoke test
npm run test:migration-smoke
```
